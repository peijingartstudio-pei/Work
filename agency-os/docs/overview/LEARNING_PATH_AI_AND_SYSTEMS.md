# 邊做邊學：不懂 AI／程式時，怎麼跟我協作、最後自己當家

> 目的：你不用先變成工程師，也能**一天比一天少依賴我**；同時把 `agency-os` 與 `lobster-factory` 變成你真正會用的系統。  
> 本檔是「長版操作手冊」，可慢慢讀；每天實作只要照 **§6 每日 15 分鐘** 即可。

---

## 1) 先對齊三件事（心態）

1. **你不用懂我在寫的每一行程式**  
   你要懂的是：**這一步在保護什麼、若失敗代表什麼、下一步該找誰／看哪份檔**。程式是工具，治理與流程才是你的本業。

2. **AI 像「極快但會錯的副手」**  
   適合：整理文件、對照規格、產生檢查清單、解釋名詞。  
   不適合：在你沒驗證前，就讓它單方面改生產設定、金鑰、計費、DNS。

3. **你的「真實來源」永遠在 repo 裡，不在聊天視窗**  
   聊天會不見；`TASKS.md`、清單、`reports/` 會留下來。養成習慣：**重要結論要回到檔案**（你自己改或請我改，但結果要進檔）。

## 2) 用兩條軸來學（不要混在一起）

| 軸 | 是什麼 | 你該會什麼程度 |
|----|--------|----------------|
| **Agency OS** | 公司怎麼運作：客戶資料夾、合約、SOP、收工檢查、健康報告 | **大師級**：不靠 AI 也能帶新案、收工、查報告 |
| **Lobster Factory** | 網站／租戶「工廠」：資料庫設計、manifest、自動化閘道 | **大師級**：懂流程與風險，**關鍵步驟能自己跑指令並解讀結果**；深程式可慢慢來 |

先精通 **Agency OS 每日迴圈**，再攻 **Lobster 的驗證閘道**；兩邊都會後，你自然會看懂兩者怎麼接起來。

## 3) 跟我協作時，請固定用這幾句（邊做邊學）

每次開一個小任務時，你可以**直接複製贴上**（可改主題）：

1. **「用國中生能懂的話，說明這一步在幹嘛，以及做錯會怎樣。」**  
2. **「列出我現在需要決定的 3 個選項（優缺點各一行）。」**  
3. **「做完之後，我要在 repo 的哪個檔案留證據？若沒有，請幫我建議一句要寫進 WORKLOG 的話。」**  
4. **「請不要直接改超過 N 個檔；若超過，先給我計畫再動手。」**（N 你自己訂，例如 3）  
5. **「請用編號步驟回我，每一步一行，最後一行是『你如何自己檢查做對了』。」**

這樣即使你看不懂 code，你仍會拿到：**目的、風險、你的決策點、驗收方式**。

## 4) 名詞超短字典（看到就查這裡）

- **Gate／閘道**：一道「不過就不能往下」的檢查（像品管關卡）。  
- **Manifest**：給「工廠」的安裝／設定清單（像菜單，告訴系統要裝哪些東西）。  
- **Dry run／乾跑**：模擬跑流程，**預設不真的寫入**資料庫或改現場。  
- **Critical Gate**：健康檢查裡**硬性的那一關**，FAIL 就代表有缺檔、路徑錯、或文件斷鍊等，要先修。  
- **Monorepo**：同一個大資料夾底下有多個子專案（例如 `agency-os` 和 `lobster-factory` 並列）。  
- **單一真實來源**：待辦狀態只信 `TASKS.md`（全公司）＋龍蝦主清單（工廠），其它地方不要各寫一份。

## 5) 四週漸進路線（可依你步調拉長）

### 第 1 週：只學「公司怎麼每天轉」

- 讀：`docs/overview/EXECUTION_DASHBOARD.md`（一頁）  
- 每天做：`docs/operations/end-of-day-checklist.md` 裡的**必跑三步**（doc-sync / health / guard）  
- 學會看：`reports/health/health-*.md` 裡 **Critical Gate PASS/FAIL**  
- 產一次綜合報告：`scripts/generate-integrated-status-report.ps1` → 開 `reports/status/integrated-status-LATEST.md`  
- **本週目標**：你能向別人解釋「為什麼收工要跑這三步」。

### 第 2 週：學「客戶與專案資料放哪」

- 讀：`tenants/NEW_TENANT_ONBOARDING_SOP.md` + `tenants/company-a/` 結構  
- 做：用紙上或副本**演練**一次新客戶資料夾要齊哪些檔（不必真客戶）  
- 對照：`TASKS.md` 里「新客戶實跑」那一條，寫下你卡在哪  
- **本週目標**：你能獨立找到「某客戶的合約／站點／專案」對應路徑。

### 第 3 週：學「龍蝦工廠的驗收語言」

- 讀：`lobster-factory/README.md` 的「建立/驗證」、`docs/C1_EXECUTION_RUNBOOK.md`  
- 跑（在有人陪跑第一次最好）：  
  - `D:\Work\scripts\verify-build-gates.ps1`（一鍵工程+治理）  
  - 或 `lobster-factory` 里 `npm run validate`（僅工程基線）  
- 學會分：**乾跑 vs 真寫入**（沒開環境變數就不該寫 DB）  
- **本週目標**：你能說出「我現在跑的是檢查還是真的會改資料」。

### 第 4 週：學「斷線時怎麼自己接起來」

- 用關鍵字 **`AO-RESUME`** 開新對話，對照 `memory/CONVERSATION_MEMORY.md` 是否一致  
- 自己更新：`TASKS.md` 一條 Next、`WORKLOG.md` 三行今天做了什麼  
- 試著**不看 AI**：只憑 `EXECUTION_DASHBOARD` + `integrated-status-LATEST` + `TASKS` 安排一天  
- **本週目標**：你有一天主要靠自己開檔、跑腳本、看報告完成工作。

## 6) 每天 15 分鐘儀式（最省力版本）

1. 打開 `reports/status/integrated-status-LATEST.md`（若過期就先跑產報腳本）。  
2. 打開 `TASKS.md`，只圈 **1 件**今天要做的事。  
3. 做完後寫 `WORKLOG.md` **3 行**（事實，不要寫心情小說）。  
4. 若今天動到治理／docs：跑 doc-sync；收工前跑 health（Critical Gate 要 PASS）。

**每週一次（與每日分開）**：執行 `scripts/weekly-system-review.ps1`——一次跑滿工程＋治理閘道、刷新綜合報告，並把證據附錄到 `WORKLOG`（見 `EXECUTION_DASHBOARD.md`「每週」一節）。若要自動化： `automation/REGISTER_WEEKLY_SYSTEM_REVIEW_TASK.ps1`。

## 7) 怎麼知道自己「已經大師級」

**Agency OS 大師**（約當）：

- 能獨立完成新客戶 onboard 的檔案與 SOP，並知道證據放哪。  
- 能讀懂 health 報告 FAIL 的類別（缺檔、路徑、編碼、地圖不一致）。  
- 收工三步能自己跑完並在 daily note 留報告檔名。

**Lobster 大師**（務實定義，不要求你寫框架）：

- 能說清楚 Phase 1 邊界：什麼已自動檢查、什麼還沒接真機。  
- 能自己跑 `verify-build-gates` 或 `bootstrap-validate`，並解讀「PASS = 什麼保證了」。  
- 知道 **真寫入**要先有什麼環境與批准，不會誤觸 production。

- 本檔路徑：`docs/overview/LEARNING_PATH_AI_AND_SYSTEMS.md`（隨 repo 走）。

## 9) 給完全外行：AI、Coding、這個 repo 各是什麼（三分鐘版）

- **AI（像我這樣的助手）**：讀你的文字與檔案、產生建議或改檔。**不握有你的電腦**，除非你允許工具執行；也**不保證永遠正確**。你要把自己想成「老闆／專案負責人」，AI 是「寫手＋顧問」，**最後盖章的是你**。

- **Coding**：多半指「用程式語言寫出可重複執行的步驟」。在你的系統裡，很多已經變成**腳本**（按一下就跑檢查）。你不必會寫，但要慢慢會：**敢跑、會看成功／失敗、知道失敗時不要慌著上 production**。

- **這個 repo（專案資料夾）**：Agency OS 與龍蝦工廠的**說明書＋清單＋自動檢查**都放在這裡。大師不是背下所有檔名，而是**永遠知道「真相」在哪幾個檔與幾份報告**。

## 10) 三層能力（你可以對照自己現在第幾層）

| 層級 | 你會做什麼 | 跟 AI 的關係 |
|------|------------|----------------|
| **L1 操作** | 會開儀表板、會跑收工腳本、會看 PASS/FAIL | AI 幫你解釋第一次，你重複到熟 |
| **L2 治理** | 會維護 `TASKS`、客戶資料夾、知道證據與日誌放哪 | AI 幫你起草，你**改狀態與決策** |
| **L3 工程判斷** | 會區分乾跑／真寫入、知道 Phase 1 邊界、會讀綜合報告排優先 | AI 幫深挖，你**做 go/no-go** |

目標是 **L1 → L2 先穩**，再拉 **L3**；不要求變成全端工程師才算做完。

## 11) 每次任務的「學習合約」（建議照抄）

開任何一段協作前，你可以在心裡或訊息里確認四件事：

1. **這次成功長什麼樣子？**（一句話）  
2. **我可以接受的最壞情況是什麼？**（例如：只壞測試環境、可還原）  
3. **證據會留在哪？**（`WORKLOG`？`reports/`？`TASKS` 勾選？）  
4. **這次我要刻意記住的一個名詞／一步驟是什麼？**（只選一個，避免貪多）

結束後用兩個問題覆盤（可問我，也可自己寫在日記）：

- 「若明天沒有 AI，我能不能靠檔案與腳本重做一次？」  
- 「哪一步我還不敢自己做？」——把那一條寫進 `TASKS` 的 Next，縮小範圍再練。

## 12) 自己審查「AI 有沒有搞砸」的最小清單

當有改動時（不論是你或我改的）：

1. **有沒有動到機密？**（金鑰、token、客戶隱私是否出現在 repo）——有的話先撤掉再談其它。  
2. **健康檢查還過嗎？**`system-health-check.ps1` → **Critical Gate PASS**。  
3. **待辦有沒有分裂？**只維護 `TASKS.md` + 龍蝦主清單，不要在第三個地方另寫一套。  
4. **你能不能向夥伴用一句話講清「改了什麼、為什麼」？**講不出來代表你還沒吸收，適合再問一次「用國中生版解釋」。

## 13) 龍蝦工廠：外行最該先記住的五件事

1. **預設是安全的**：很多驗證是**檢查與乾跑**，不會自動幫你開 DB 寫入。  
2. **真寫入是「顯式開關」**：要有環境變數與你的意圖，沒有就不要假設已寫入。  
3. **Manifest**：「這次要裝／做什麼」的清單；變更 manifest 通常要過 **validate**。  
4. **`verify-build-gates.ps1`**：工程 + 治理一鍵；適合你每週或送 PR 前跑一次。  
5. **主清單**：`lobster-factory/docs/LOBSTER_FACTORY_MASTER_CHECKLIST.md` — 看不懂細節沒關係，先看 **A 區到哪、C 區卡哪**。

## 14) Agency OS：外行最該先記住的五件事

1. **EXECUTION_DASHBOARD**：每天一頁入口。  
2. **收工三步**：doc-sync → health → guard（意義是：文件、骨架、狀態都有留痕）。  
3. **`tenants/`**：一個客戶一包；大師要知道**新客戶從哪個 SOP 開始**。  
4. **報告目錄**：`reports/health`、`reports/closeout`、`reports/guard` — 是「發生過什麼」的憑證。  
5. **`AO-RESUME` / `AO-CLOSE`**：不是魔法，是強迫大家**先讀記憶與任務**再做事，降低断線成本。

## 15) 三個月學習地圖（可拉長到半年）

- **月 1**：只做熟 **Agency OS 每日迴圈 + 綜合報告 + TASKS/WORKLOG 習慣**；龍蝦只跑到「能成功執行一鍵閘道、看懂 PASS」。  
- **月 2**：**演練新客戶 onboard**（可用假資料）；龍蝦讀完 **C1 runbook**，自己跑一遍**乾跑與 validate**（有人旁觀亦可）。  
- **月 3**：挑 **一條**龍蝦主線（例如：從乾跑到「測試環境真寫入 workflow_runs」若你環境允許）**寫成你自己的檢核表**；同時試 **24～48 小時刻意少用 AI**，只靠檔案與腳本。

## 16) 你不用我之後，「大師」的一天長什麼樣子（寫實版）

- 早上：**monorepo 根**先 **`git pull --ff-only origin main`**（雙機必做；見 `REMOTE_WORKSTATION_STARTUP`）→ 開 `EXECUTION_DASHBOARD` → `integrated-status-LATEST` → `TASKS` 選一件主線 → 打 **`AO-RESUME`**。  
- 做事：客戶與交付走 `tenants` + 專案慣例；工程變更只進**有 gate** 的流程。  
- 傍晚：`WORKLOG` 三行；若改 docs 則 doc-sync；health Critical PASS；guard 更新狀態。  
- 週期性：龍蝦清單勾選與 repo push，確保**另一台電腦 pull 也活得下去**。

## 17) 要注意什麼、邏輯怎麼推演、為什麼這樣做、效益與差錯、怎麼檢查與預防

> 這一節回答：「除了照做，我腦子裡該跑哪條邏輯？」**會**更好——因為你從「背步驟」變成「能判斷步驟還值不值得信任」，將來才能離開 AI 獨立決策。

### 17.1 通用：每次遇到新流程／新腳本／請 AI 改 repo，都用「五連問」

| 順序 | 問題 | 你在找什麼 |
|------|------|------------|
| 1 **注意** | 這一步**假設了什麼**才成立？（環境、權限、路徑、誰已經先做過什麼） | 隱藏前提；前提錯了，後面全錯 |
| 2 **推演** | 若照做成功，**下一個必然發生什麼**？若失敗，**最早會在哪裡爆**？ | 因果鏈；知道去哪裡查 log／報告 |
| 3 **為什麼** | 若不這樣做，**省下的成本是什麼、換來的風險是什麼**？ | 制度不是儀式，是風險對價 |
| 4 **效益／差錯** | **做對**帶來什麼（可驗收）？**做錯或跳過**典型下場是什麼？ | 你才知道值不值得堅持 |
| 5 **檢查／預防** | **怎麼驗收**（PASS 長什麼樣）？**怎麼預防再犯**（習慣、鎖、清單）？ | 把「相信」變成「可證明」 |

口訣：**前提 → 因果 → 取捨 → 對錯後果 → 驗收與預防**。

### 17.2 套用：`TASKS.md` + 龍蝦主清單（單一真實來源）

- **注意**：不要在第三個地方（聊天、腦袋、隨機備忘）另開一套待辦；一變兩套就會**分歧**。  
- **推演**：你改 `TASKS` 勾選 → 明天開檔的人只看得到這裡 → **敘事與事實對齊**。  
- **為什麼**：多處維護 = 一定會漏改一處 → 誤判進度 → 錯誤承諾。  
- **效益**：永遠知道「官方進度」在哪；**差錯**：雙軌待辦 → 重工、漏做、吵架。  
- **檢查**：綜合報告 §1–2 與 `TASKS` 未勾項是否一致；**預防**：新想法先寫進 `TASKS` 再動手。

### 17.3 套用：收工三步（doc-sync → health → guard）

- **注意**：改的是「治理／文件／連動」時，doc-sync 才有意義；純寫程式未動 docs 時，重點在別的 gate。  
- **推演**：doc-sync 產 closeout 報告 → health 掃結構與連結 → guard 寫狀態／告警 → **隔天 AO-RESUME 有憑有據**。  
- **為什麼**：人會忘、聊天會不見；**機器可重跑的檢查**留下時間戳與 PASS/FAIL。  
- **效益**：斷線、換電腦、半年後回來仍能接；**差錯**：跳過 → 以為系統健康、其實缺檔或斷鍊已擴散。  
- **檢查**：`Critical Gate PASS`、報告檔名寫進 `memory/daily`；**預防**：收工清單固定勾、不要「累就先跳」。

### 17.4 套用：`verify-build-gates.ps1`（工程 + 治理一鍵）

- **注意**：要在對的 repo 根（`D:\Work` monorepo）；`agency-os` 與 `lobster-factory` 要都在預期位置。  
- **推演**：bootstrap（含 manifest／governance／doc 連結）→ 再跑 health（租戶／地圖／排程等）→ **兩類問題都會浮上來**。  
- **為什麼**：工程對、治理斷鍊仍會害營運；反之亦然。  
- **效益**：送 PR、大改 docs、換機前**一次看清**；**差錯**：只跑一半 → 假安全。  
- **檢查**：終端機 exit 0 + 報告路徑有輸出；**預防**：大改前後固定跑、寫進 `WORKLOG` 一行。

### 17.5 套用：龍蝦「乾跑」vs「真寫入」

- **注意**：有沒有**顯式**開啟寫入相關環境變數／參數；沒有就當**沒寫進 DB**。  
- **推演**：乾跑只看 payload／模板 → 真寫入會碰實際資料 → **錯誤成本差一個量級**。  
- **為什麼**：預設安全是為了防止「誤跑一次就污染資料／客戶環境」。  
- **效益**：可重複演練；**差錯**：以為在測試其實寫 production、或反過來以為寫了其實沒寫。  
- **檢查**：看腳本說明與 runbook、在 Supabase／日誌**對帳**一筆；**預防**：測試專用專案／權限分離、變更前兩人確認（若你團隊有第二人）。

### 17.6 與 AI 協作時，額外要注意的「差錯型態」

| 類型 | 典型情況 | 預防／檢查 |
|------|----------|------------|
| **幻覺** | AI 說某檔存在或某指令一定對 | 以 repo 與腳本輸出為準；跑 health／validate |
| **過度修改** | 一次改太多檔，難以 review | 限制 N 個檔、要求先給計畫（見 §3） |
| **機密外洩** | 金鑰被貼進 md 或 log | 機密永不進 git；定期搜尋 `key`、`token` 模式（或專用掃描） |
| **與真實來源打架** | 聊天裡的進度與 `TASKS` 不一致 | 以 `TASKS`／清單為準；聊完勾一筆 |
